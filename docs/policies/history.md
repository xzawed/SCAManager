### 정책 7 강화 (2026-05-02 Phase 1 회고 후속): PR 단위 = 응집 단위 분할

기존: "모든 작업은 PR 단위. 가급적 main 직접 작업 지양."

**강화**:
- **PR 단위 = 응집 단위 분할** 명시 ("작게 분할" 아님)
- **응집 = URL + 화면 + 데이터 3종 동시 묶음** (2026-05-02 Phase 1+2 회고 P0 #1) — 신규 기능 도입 시 (a) URL/route, (b) 화면/template, (c) 데이터/service 3종을 같은 PR 으로 묶음. 하나라도 분리되면 사용자 진입 경로 깨짐.
- **사용자 인지가 깨지는 변경은 동일 PR 의무**:
  - 신규 라우트 + 기존 URL redirect = 동일 PR
  - 신규 라우트 + nav 링크 갱신 = 동일 PR
  - 라우트 폐기 + 폐기 라우트로의 외부 링크 정리 = 동일 PR
- **비응집 분할 사례** (Phase 1 사례):
  - PR 4 (신규 /dashboard) ↔ PR 5 (redirect + nav) 분리 → nav 일시 404
- **응집 분할 사례** (정정 default — Phase 2+ 검증):
  - Phase 2 PR 1 = "Auto-merge 카드 1세트" (service + 라우트 + 템플릿 + 테스트 동일 PR) → nav 깨짐 0건

  - Supabase 호환 fix = "DB 환경 호환" (SQL + runbook + CLAUDE.md docs 동일 PR)

#### (1) 강화 (사이클 84 — i18n 18 PR 회고 Tier B Q2 사용자 OK): "단일 작업일 dispatch 횟수" vs "누적 agent invocation" 구분 명시

🔴 사이클 84 = dispatch **2회** (기획안 5+1 + Phase 1 사전검토 3 에이전트) 만 발화 but 누적 agent invocation = **9회** (5+1 = 6 + 사전검토 3). dispatch 횟수 vs invocation 횟수 비대칭으로 정책 8 진화 (1) 임계 적용 영역 모호.

**본 강화**:
- **dispatch 횟수** = 단일 메시지 내 Agent tool 호출 그룹 (5+1 = 1 dispatch / 3 에이전트 병렬 = 1 dispatch)
- **agent invocation** = 개별 Agent tool 호출 합 (5+1 = 6 invocation / 3 에이전트 = 3 invocation)
- **임계 적용**: 본 의무 (1) "≥ 5회" = **dispatch 횟수** 기준 default. **단, 누적 invocation ≥ 30 시 동등 사전 확인 의무 적용** (사이클 78~82 36 invocation 누적 최고치 패턴 회귀 가드).
- 정책 6 강화 (line:span 실측 의무) 페어 — invocation 카운트 추정 X 실측 의무 (Agent tool 호출 카운트 직접 회신).

#### (2) 개별 PR cross-verify 효과 commit body 정량 명시 의무 (사이클 70~72 회고 P0-7 명시화)

5+1 cross-verify 진행 PR commit body §"cross-verify 결과" 섹션 의무 — false-positive 차단 N건 / 신규 발견 N건 / Tier A 정정 N건 정량 명시. 회고 보고서 부재 PR 도 ROI 추적 가능. 본 진화 = 정책 6 강화 (line:span 실측 의무) 페어.
